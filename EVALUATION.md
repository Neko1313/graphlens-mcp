# Оценка проекта graphlens-mcp

Дата: 2026-06-26. Оценка по состоянию ветки `claude/project-evaluation-gaps-qcbk27`
(HEAD `3cf5c7c`). Проверено: чтение всего `src/`, тестов, CI; запуск `ruff`, `ty`,
`pytest`, а также прямое воспроизведение находок в SQLite/aiosqlite.

## Краткое резюме

Идея и архитектура сильные: тонкий stateful-слой (хранилище + свежесть + surface для
агента) над stateless-движком `graphlens`, аккуратные инварианты (стабильные id, нормализация
путей, file-owned writes, cycle-safe CTE), типизированный контракт MCP-инструментов, чистый
проход `ruff (ALL)` и `ty`. Документация (README, ARCHITECTURE) подробная и честно перечисляет
ограничения.

Но в **закоммиченном** состоянии присутствует критический дефект, из-за которого сервер
вообще не запускается, а тестовый набор зависает. После его устранения (см. ниже) 22 unit-теста
проходят. Это указывает и на пробел в CI/процессе: showstopper-баг дошёл до HEAD.

## Критично

### 1. Загрузчик схемы сломан — хранилище не открывается, сервер не стартует
`store/sqlite_store.py::_apply_schema` разбивал `schema.sql` наивно через `schema.split(";")`.
В `schema.sql` (строка 28) внутри SQL-комментария есть литеральный `;`:

```
-- reject such "dangling" edges; instead we keep them and filter unresolved targets
```

Сплиттер режет комментарий пополам, и следующий «оператор» начинается со слова `instead` →
`sqlite3.OperationalError: near "instead": syntax error`. Воспроизведено напрямую:

- корректный `executescript` создаёт все 6 таблиц + FTS5;
- наивный `split(";")` падает и создаёт только `files, meta, nodes` — **без** `edges`, `deps`,
  всех индексов и виртуальной таблицы `nodes_fts`.

Последствия: `SqliteStore.create` падает при **любой** команде (`init`/`serve`/`status`/
`reindex`) — проект нефункционален. Дополнительно `aiosqlite` после ошибки в worker-потоке
подвешивает процесс (поэтому `pytest` не завершался, а висел).

**Статус: исправлено в этой ветке** — `_apply_schema` переведён на `conn.executescript(schema)`,
который корректно разбирает комментарии и границы операторов. После фикса `pytest tests/unit`
зелёный (22 passed).

## Высокий приоритет

### 2. Пробел в CI/процессе: showstopper дошёл до HEAD
Баг (1) — ровно тот класс, что ловится `test_store`. Раз он в HEAD, значит pytest в CI либо не
был зелёным, либо завис/таймаутил. Сопутствующее: в `.github/workflows/ci.yml` нет
`timeout-minutes` у job — зависание (как при этом баге) сожжёт весь лимит раннера вместо быстрого
падения. Рекомендация: добавить `timeout-minutes`, а также `pytest-timeout` с per-test лимитом,
чтобы один зависший оператор не вешал весь прогон.

### 3. `resolver_status` отражает только файл запрошенного узла, а не возвращённых
README/ARCHITECTURE обещают: «каждый ответ несёт graph-quality, чтобы агент не принял частичный
результат за полный». Но `server/tools.py::_fresh_status` проверяет свежесть/статус **только**
файла `node_id`. `get_callers`/`get_callees`/`find_references` могут вернуть узлы из файлов со
статусом `skeleton`, при этом отрапортовав `ok`. Агент сделает вывод «полно», хотя часть рёбер
непросчитана. Симптоматично, что метод `SqliteStore.get_worst_status_for_files`, написанный
ровно для агрегации статуса по множеству файлов, **нигде не вызывается** — мёртвый код, который и
должен был закрывать эту дыру. Рекомендация: агрегировать статус по `file_path` всех возвращаемых
узлов (через уже существующий helper).

## Средний приоритет

### 4. Инкрементальный `apply_patch` вставляет ВСЕ узлы графа, а не только принадлежащие файлу
В `apply_patch` рёбра и FTS фильтруются по `owned_ids`/`file_path == file_path`, но цикл вставки
узлов — `for node in graph.nodes.values()` — не фильтруется. Одно­файловый `analyze()` может
эмитить внешние/stub-узлы, и они апсёртятся по стабильному id (`ON CONFLICT DO UPDATE`),
перезаписывая `file_path`/`span`/`kind` у символов, принадлежащих другим файлам. Полный индекс
этого избегает (`subgraph_for_file`), инкрементальный — нет. Рекомендация: ограничить вставку
узлами с `node.file_path == file_path` (как уже сделано для рёбер/FTS).

### 5. Нет отката транзакции при ошибке внутри патча
`apply_patch`/`apply_structural`/`delete_file` держат `_write_lock`, выполняют несколько
`execute` и затем `commit`, но без `try/except → rollback`. Если оператор посередине упадёт,
частичные изменения остаются в незакоммиченной транзакции соединения и могут быть закоммичены
следующим писателем. Рекомендация: обернуть тело в `try/except` с `await conn.rollback()`.

### 6. `Workspace.close()` лезет в приватные внутренности graphlens
Очистка резолверов делает `getattr(adapter, "_resolver", None)` и перебирает методы
`shutdown/close/stop`. Это хрупкая связка с приватным API движка: его рефакторинг тихо сломает
завершение resolver/LSP-процессов → утечка процессов. Желателен публичный контракт shutdown со
стороны `graphlens` (или хотя бы тест, фиксирующий текущее поведение).

### 7. Документация противоречит коду
- README → «Known limitations»: *«No `remove`/`clean` command yet»*, но команда `remove`
  полностью реализована в `cli.py` и описана в ARCHITECTURE. Таблица «Commands» в README её тоже
  не упоминает.
- `get_node_info` (docstring и README) обещает «signature + docstring», но инструмент возвращает
  лишь сырой срез исходника (`_read_span`); отдельного извлечения сигнатуры/докстринга нет, и в
  `NodeRef`/`NodeInfoResult` нет соответствующих полей.

## Низкий приоритет / полировка

8. **Транзитивная свежесть** — задокументированное ограничение: если изменился файл-вызыватель, а
   запрошен только целевой узел, impact-анализ может быть устаревшим. Приемлемо для v1, стоит на
   roadmap.
9. **Один общий коннект сериализует чтения за записями** — осознанный компромисс; для одного
   пользователя ок, но ограничивает пропускную способность (отдельный reader-коннект напрашивается).
10. **`full_index` последователен по языкам** несмотря на семафор `MAX_CONCURRENT_RESOLVERS=4`
    (семафор реально работает лишь на инкрементальных запросах). Языки можно индексировать
    параллельно.
11. **`_synthesize_cross_language_edges` — O(exposers × consumers)** на каждый boundary и
    материализует обе стороны; на hub-boundary с многими потребителями это может разрастись.
12. **Покрытие тестами** end-to-end есть только для Python. Не покрыты: кросс-языковые рёбра
    (TS/Go/Rust/PHP), дедуп `InFlightRegistry`, пересборка по schema-fingerprint, агрегация
    `resolver_status` (см. п.3).
13. **Мелочи идентичности**: email автора в `pyproject.toml` (`...@yandex.ru`) не совпадает с
    git-конфигом; версия `0.1.0`, README честно помечает статус «early».
